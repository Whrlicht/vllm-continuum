# Trace Replay 实验复现报告（Continuum vs vLLM）

## 1. 实验说明

本报告基于以下 6 组复现实验结果（并发度 32/48/64）进行对比分析：

- llama3.1-8b-continuum-trace_replay-fixed-32
- llama3.1-8b-continuum-trace_replay-fixed-48
- llama3.1-8b-continuum-trace_replay-fixed-64
- llama3.1-8b-vllm-trace_replay-fixed-32
- llama3.1-8b-vllm-trace_replay-fixed-48
- llama3.1-8b-vllm-trace_replay-fixed-64

数据来源均为各目录下的 results.json。每组均为 300 jobs。

## 2. 核心结论（先看结论）

1. Continuum 在三档并发下，整体 E2E 延迟和尾延迟（average / p95 / p99）均优于 vLLM。
2. Continuum 吞吐也整体更高：在并发 64 时优势更明显（output throughput +10.69%）。
3. 但 Continuum 在中高并发出现明显调度副作用：
   - 抢占（preempted）显著高于 vLLM。                    [#]
   - 等待队列深度（num_requests_waiting）显著增加。       [#]
   - TTFT 分布更不稳定：虽然 p95 更好，但 mean 和极值（尤其 max）更差，存在少量“超慢首 token”请求。
4. 结论上，Continuum 当前是“平均更快、吞吐更高，但稳定性和极端尾部风险更大”。

## 5. 重点分析：Continuum 当前不足与缺点

## 5.1 首 token 延迟分布不稳（最重要）

现象：

- 并发 48/64 下，Continuum 的 ttft_mean 反而比 vLLM 高。
- 并发 64 时，ttft_max 达到 296s（vLLM 仅约 29s），ttft_p99 也更差（24.37s vs 14.34s）。
- 但 ttft_p95 又优于 vLLM。

解释：

- 说明 Continuum 的大多数请求首 token 更快，但少数请求被“拖得很慢”，形成肥尾。
- 这是典型的调度公平性与吞吐优化冲突：策略偏向整体效率，牺牲了小部分请求体验。

风险：

- 在线服务场景下，个别请求超时会直接影响可用性和用户感知。
- 平均值优秀不能掩盖 SLO 违规风险。

## 5.2 抢占频率显著偏高

现象：

- 并发 48/64 下 preempted_mean 远高于 vLLM。
- 并发 64：Continuum preempted_mean=0.657，vLLM=0.027；p95=5 vs 0。

解释：

- Continuum 可能使用了更激进的请求切换来换取吞吐与总体延迟。
- 过高抢占会带来状态切换开销、请求抖动与尾部放大。

风险：

- 增加调度抖动，削弱可预测性。
- 可能导致 TTFT 极端值恶化（与 5.1 相互印证）。

## 5.3 等待队列在高并发下明显堆积

现象：

- 并发 64：waiting_mean 1.736 vs 0.562，waiting_p95 14 vs 4。
- 并发 48：waiting_mean 0.551 vs 0.179，waiting_p95 5 vs 1。

解释：

- Continuum 在中高负载下调度“挤压”更明显。
- 大部分请求可能仍能快速推进，但部分请求等待时间会被放大。

风险：

- 负载进一步上升时，肥尾与超时风险可能非线性增长。

## 5.4 高并发下等待气泡（waiting bubble）恶化

现象：

- 从 job_round_metrics 聚合看，并发 64 的 avg_wait_bubble：
  - Continuum: 4.307
  - vLLM: 1.131

解释：

- 说明 Continuum 在轮次衔接或资源争用中存在更长空转等待。
- 该问题通常和抢占策略、批次组装策略、KV 资源竞争相关。

## 5.5 可解释性指标缺口（工程层面缺点）

现象：

- 当前结果虽有 scheduler_stats / iteration_stats，但报告里缺少“按请求类别/长度桶/轮次类型”的归因拆解。

风险：

- 只能看到“结果”，难定位“为什么慢、慢在哪一类请求”。
- 优化迭代会变慢，容易靠经验调参。

## 6. Continuum 的改进建议（按优先级）

## P0：控制极端 TTFT（先保稳定性）

建议：

- 在调度器中加入 anti-starvation 机制：
  - 对等待过久请求提升优先级（aging）。
  - 设置 TTFT-aware 的最大等待轮数阈值。
- 增加 tail-guard：
  - 当请求等待超过阈值时，禁止再次被抢占或强制插队一次。
- 把 SLO 目标从“均值导向”改为“分位导向”：至少纳入 ttft_p99 和 ttft_max 约束。

预期收益：

- 在吞吐小幅波动下，大幅降低极端慢请求概率。

## P0：降低高并发下抢占率

建议：

- 对活跃轮次设置最小运行配额，减少来回切换。
- 引入 preemption budget（单位时间最大抢占次数）。
- 在高负载时动态切换到更保守策略（例如降低抢占阈值敏感度）。

预期收益：

- 降低调度抖动和状态切换成本，改善 tail 和一致性。

## P1：缓解队列堆积与等待气泡

建议：

- 细化 batch 组装策略：按 prompt 长度 / 估计 decode 轮数分桶，减少长短任务互相阻塞。
- 对大 prompt 或超多轮任务做 admission control / 并发上限。
- 利用 iteration_stats 做“空转周期”定位，针对性优化调度步长和批次刷新频率。

预期收益：

- 在并发 48/64 下收敛等待队列，减少波动。

## P1：改进观测与归因能力

建议：

- 增加以下指标分桶输出：
  - 按 prompt 长度、round 数、是否高 KV 命中。
  - 每请求 preempt 次数、累计等待时间、首次调度延迟。
- 在可视化中新增：
  - TTFT tail 请求 TopN 的调度轨迹。
  - preemption 热点时间窗与请求类型关联图。

预期收益：

- 快速定位“谁拖慢了系统”，避免盲目调参。

## P2：策略参数自动化寻优

建议：

- 以并发度和负载结构为条件，自动选择调度参数（阈值、配额、抢占预算）。
- 通过离线 trace replay 搜参，目标函数使用加权多目标：
  - maximize throughput
  - minimize e2e_p95/p99
  - minimize ttft_p99
  - minimize preempted_mean

预期收益：

- 不同工作负载下更稳定地逼近 Pareto 最优。

## 7. 补充观察

1. 两个系统的 KV reuse rate 均接近 1，说明缓存复用机制总体有效。
2. Continuum 的 prefix hit rate 略低于 vLLM（差距不大），不是当前主要瓶颈。
3. Continuum 在并发升高时吞吐下降幅度略小于 vLLM，表明其扩展性方向是正确的，但需要补齐稳定性。

## 8. 结论

如果目标是“吞吐和总体延迟”，当前结果支持 Continuum 优于 vLLM。

如果目标是“线上稳定性和尾部体验”，Continuum 还存在明显短板，尤其是中高并发下的 TTFT 极端值、抢占过高、等待堆积。

建议优先推进 P0 级优化（反饥饿 + 抢占预算 + TTFT tail guard），再进行 P1/P2 的结构化迭代。
